# BIP158: Compact Block Filters for Light Clients

## Overview

**BIP158** introduces compact block filters for Bitcoin light clients. It allows clients to efficiently determine if a block contains transactions relevant to them (e.g., sending to their addresses) without downloading the entire block.

**Purpose**:
- Reduce bandwidth for SPV (Simplified Payment Verification) clients
- Maintain privacy by not revealing which addresses the client watches
- Enable efficient wallet synchronization
- Provide alternative to full node while maintaining reasonable privacy

**Status**: Implemented in Bitcoin Core since v0.19.0 (released November 2019)

---

## How BIP158 Works

### 1. Filter Construction

For each block, a **Golomb-coded set (GCS)** filter is created containing:

**Elements included in filter**:
- All transaction outputs' `scriptPubKey`
- All transaction inputs (excluding coinbase transactions)
- Block headers from previous blocks (for efficient chain validation)

**Block 1 block**: First 6 bits of scriptPubKey hash (SHA-256) and 2-byte block height

### 2. Golomb-Rice Coding

Filters use **Golomb-Rice coding** for compression:

**Algorithm**:
1. Hash each element with SipHash-2-4 (using block hash as key)
2. Divide hash range into buckets (parameter P, typically 20)
3. Store quotient (unary) and remainder (binary)

**Benefits**:
- High compression (~50 bytes per filter)
- Fast lookup (~1ms)
- Configurable false positive rate

### 3. Filter Properties

| Property | Value | Description |
|----------|-------|-------------|
| Filter size | ~50 bytes | Per block (average) |
| False positive rate | ~0.001% | 1 in 100,000 |
| Lookup time | ~1ms | Per filter check |
| Block header size | 80 bytes | Per block |

**Storage requirements**:
- Full chain: ~50 MB (vs 600 GB for full blocks)
- 100 blocks: ~5 KB (vs 1-2 GB for full blocks)

## Implementation Options

### Option 1: Bitcoin Core (C++)

Bitcoin Core includes full BIP158 implementation since v0.19.0.

**Usage**:
```cpp
// Generate filter for a block
CBlock block;
CBlockFilter filter(block, BlockFilterType::BASIC);

// Check if element matches filter
GCSFilter::ElementSet elements = {scriptPubKey};
bool matches = filter.GetFilter().MatchAnyElement(elements);

// Serialize/deserialize filters
CDataStream ss(SER_NETWORK, PROTOCOL_VERSION);
ss << filter;
```

**Access via RPC**:
```bash
# Get block filter for a specific block
bitcoin-cli getblockfilter <blockhash> "basic"

# Output:
# {
#   "filter": "hex_encoded_filter",
#   "header": "filter_header_hash"
# }
```

**Requirements**:
- Bitcoin Core v0.19.0 or later
- Full node sync (~600 GB for mainnet)
- Enable `-blockfilterindex` flag

**Configuration**:
```bash
# bitcoin.conf
blockfilterindex=1
txindex=1
```

## BIP158 vs PIR for BitcoinPIR Project

### Similarities

Both approaches provide privacy when querying blockchain data:

| Feature | BIP158 | PIR |
|---------|--------|-----|
| Privacy goal | Hide which addresses you watch | Hide which blocks you query |
| Server trust | Server must be honest (can lie about filter matches) | Server cannot learn which block you access |
| Bandwidth | ~50 KB per 1000 blocks | Similar (depends on PIR scheme) |
| Computation | Client-side filtering | Server-side computation |

### Key Differences

**BIP158**:
- ✅ Implemented and production-ready
- ✅ Standard Bitcoin protocol
- ✅ Faster lookup (1ms vs 10-100ms for PIR)
- ❌ Requires downloading filter headers first
- ❌ False positives (0.001%)
- ❌ Server can lie about matches

**PIR** (Information-Theoretic or Computational):
- ✅ Information-theoretic privacy (server learns nothing)
- ✅ No false positives
- ✅ No pre-downloaded data needed
- ❌ Slower (10-100ms per query)
- ❌ More complex implementation
- ❌ Not standard protocol

### Which to Choose?

**Use BIP158 if**:
- You need production-ready solution
- You can download filter headers (~5 MB)
- You trust server not to lie about matches
- Performance is critical

**Use PIR if**:
- You need information-theoretic privacy
- You cannot pre-download data
- You don't trust server
- False positives unacceptable

### Hybrid Approach

Combine both for enhanced privacy:

1. Use BIP158 filters to narrow candidate blocks
2. Use PIR to retrieve specific block
3. Verify block contains relevant transaction

**Benefits**:
- Fewer PIR queries (only relevant blocks)
- Reduced bandwidth
- Server learns less than PIR alone

## Integration Example: Light Client Wallet

```python
class LightClientWallet:
    def __init__(self, rpc_url):
        self.rpc_url = rpc_url
        self.watched_addresses = set()
        self.block_filters = {}
    
    def add_address(self, address):
        self.watched_addresses.add(address)
        script_pubkey = address_to_scriptpubkey(address)
        return script_pubkey
    
    def sync_chain(self, start_height, end_height):
        # Download filter headers for range
        filter_headers = self.download_filter_headers(
            start_height, end_height
        )
        
        # Check each block filter
        relevant_blocks = []
        for height, header in enumerate(filter_headers):
            block_hash = get_block_hash(height)
            filter = self.get_block_filter(block_hash)
            
            # Check if any watched address matches
            for addr in self.watched_addresses:
                spk = address_to_scriptpubkey(addr)
                if filter.match_any([spk]):
                    relevant_blocks.append(height)
                    break
        
        # Download only relevant blocks
        for height in relevant_blocks:
            block = self.get_full_block(height)
            self.process_block(block)
    
    def process_block(self, block):
        # Find transactions for watched addresses
        for tx in block['tx']:
            for out in tx['vout']:
                addr = extract_address(out['scriptPubKey'])
                if addr in self.watched_addresses:
                    print(f"Found payment to {addr}: {tx['txid']}")
```
